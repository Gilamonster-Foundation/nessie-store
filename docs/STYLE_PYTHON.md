# Python style guide — monty-tui

Python in this repo means the **PyO3 bindings for `monty-events`** (published to PyPI), plus any pure-Python helpers in `python/`. There is no Python TUI side — the TUI is Rust-only.

Aligned with gilabot's house style (black + ruff + mypy + pytest).

## Toolchain

- Python ≥ 3.10 (no positional-only param tricks below that).
- **Formatter:** `black` with default config. No `pyproject.toml` overrides on line length unless we change everyone's.
- **Linter:** `ruff check`. Configured in `pyproject.toml`; we use the `E,F,W,I,B,UP,SIM,PL,RUF` rule set as a starting point and explicitly opt out of rules that conflict with black.
- **Types:** `mypy --strict`. Stubs (`.pyi`) are committed alongside the PyO3 module so consumers get type hints out of the box.
- **Tests:** `pytest -n auto` (parallel). Coverage tracked via `coverage.py`; aim for ≥ 80% on shipped surfaces.

All four pass before a PR merges. CI runs them; pre-push hook runs them locally.

## Naming

- Modules: `monty_events`, `monty_events.swarm`, `monty_events.sources`. Underscores, not hyphens, on the import side. PyPI package name is hyphenated (`monty-events`); the Python module name is the underscore form (`monty_events`).
- Public functions/methods: `snake_case`. Private: leading underscore.
- Classes: `PascalCase`. Type variables: single uppercase letter (`T`, `K`).
- Constants: `SCREAMING_SNAKE_CASE`.

## Type hints

- Type hints are **mandatory** on every public function signature, including kwargs and returns.
- `from __future__ import annotations` at the top of every Python file. Modern annotation semantics, lazy evaluation, no string-escape contortions for forward refs.
- Use `typing.Annotated` for value-shape constraints when it helps the reader (`Annotated[int, "milliseconds"]`).
- Prefer `Self` over `'ClassName'` in returns.

## Errors

- Custom exceptions inherit from a single `MontyEventsError` base. Don't raise built-ins for domain failures.
- Translate Rust errors into Python exceptions in the PyO3 layer with `From<RustError> for PyErr`. The mapping is one-to-one and explicit — no catch-all `Exception::new`.
- Error messages are user-facing. Punctuation, capitalization, and a hint about *what to do next* — same bar as the Rust side.

## Async

- We use `asyncio`, not `trio` or `anyio`. If you're tempted to introduce a second async runtime, write down why.
- Synchronous functions don't *also* exist as `_async` variants unless we have a real use case for both. Pick the right shape per surface; bridge with `asyncio.run_in_executor` at the edge if needed.

## PyO3 specifics

See [STYLE_PYO3.md](STYLE_PYO3.md) for the cross-cutting Rust ↔ Python rules. Highlights for the Python side:

- The Rust `monty-events` crate exposes a `#[pymodule]`-decorated entry; `maturin develop` builds it into a local virtualenv for fast iteration.
- Stubs live in `python/monty_events/__init__.pyi`. They are generated by hand from the Rust signatures and committed; consumers shouldn't need to install Rust to get autocomplete.

## Testing

- `pytest` over `unittest`. Fixtures > class-based setUp/tearDown. Parametrize aggressively (`@pytest.mark.parametrize`).
- Mock external resources (NATS, HTTP, filesystem) at the call boundary. Tests don't talk to the real swarm; that's what integration tests in CI staging do.
- Regression test for every bug fix. The PR description names the bug; the test docstring references it.

## Dependencies

- Runtime deps land in `pyproject.toml [project.dependencies]`. Dev deps land in `[project.optional-dependencies.dev]`.
- Pin to the minor version. `pip-audit` runs in CI; security-sensitive bumps don't wait.

## Docstrings

- Every public function and class has a docstring. Google or NumPy style, consistent within the module.
- The first line is a sentence-fragment summary. Body explains parameters, returns, raises. Examples in `Examples:` section when useful.

## Imports

- Sorted by `ruff`'s `I` rule (isort-compatible). Standard lib, then third-party, then first-party (`monty_events.*`).
- No wildcard imports except in `__init__.py` re-exports — and even there, prefer explicit naming.
- No conditional imports inside function bodies unless it's an optional dep (`if TYPE_CHECKING`) — and if you do conditional import, document why.
